IMPLEMENTATION PLAN · STORY 1.2 ONLY

Detailed stock metrics

Expand the selected-stock card with daily trading metrics, fundamentals, and an accessible 52-week position indicator. Preserve useful quote data when optional Finnhub endpoints fail.

Outcome and boundaries

Scenario 1

After preview selection, show current, open, day high, day low, market cap, and P/E.

Scenario 2

Show 52-week low/high and visually locate current price within that range.

Excluded: watchlists, expanded cards, charts, visible range percentage, and Story 1.3+ work.

  • Desktop: three-column daily-metric row; two-column fundamentals row.
  • Mobile: two-column metric grid; market cap may span both columns.
  • Range uses a dot only. Assistive technology receives the computed position.
  • Loading selection replaces the old card with a compact loading card.
  • Missing fields render N/A; a warning names each unavailable field.

  1. Add strict raw Finnhub response types and nullable domain metric fields.
  2. Refactor selection to pass a typed stock preview into detail loading, avoiding a duplicate search request.
  3. Fetch quote, profile, and metrics concurrently. Quote is required; optional requests degrade independently.
  4. Translate missing, absent, non-finite, or non-positive optional values to unavailable domain values.
  5. Compute range position with a pure function, clamp to 0–100, and return unavailable for invalid/equal bounds.
  6. Keep formatting and unavailable-field derivation in small pure helpers; keep components under project limits.

Execution sequence

RED TDD walking skeleton

  1. Add only [TEST] comments for the two acceptance scenarios and supporting behavior; pause for confirmation.
  2. Replace the first comment with an App-level MSW acceptance test covering all detailed metrics.
  3. Predict compile failure, run all tests, add the smallest types/API surface needed to compile.
  4. Predict assertion failure, run all tests, then minimally map and render the metrics.
  5. Repeat the two-step red cycle for the 52-week range and accessible position.
  6. Add behavior coverage for partial optional-endpoint failures, named warnings/N/A, loading replacement, and P/E fallback only where required to stabilize the agreed contract.

GREEN Minimal implementation

  1. Introduce profile and metric clients plus concurrent orchestration.
  2. Update preview selection flow to reuse its typed symbol/company identity.
  3. Build focused metric-grid, warning, loading, and range-indicator presentation pieces as complexity demands.
  4. Use existing CSS tokens only; add semantic tokens first if the design needs a value not represented.
  5. Run the complete suite after every green step.

TIDY After all tests pass

  1. Invoke tidy; verify baseline tests are green.
  2. Scan production code for dead code, naming, methods over 25 lines, duplication, conditionals, locals, and imports.
  3. Present numbered opportunities and wait for approval. Do not alter test code during tidy.
  4. Apply approved refactors individually, running all tests after each.
