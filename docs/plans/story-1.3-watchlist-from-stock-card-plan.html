IMPLEMENTATION PLAN - STORY 1.3 ONLY

Watchlist from stock card

Add a compact watch control to the selected stock card, persist watched snapshot rows in localStorage, and expose a same-page watchlist tab so the user can verify the stock was tracked.

Interview decisions

Stock card control

A3 header action chip: [☆ Watch] inactive and [★ Watching] active.

Navigation

Same-page tabs: Search and Watchlist.

Persistence

useWatchlist() backed by localStorage.

Duplicates

Symbol is unique; adding an existing symbol updates the stored snapshot.

Scope

Included

  • Add/remove toggle from selected stock card.
  • Active visual state when stock is already watched.
  • Status message for add and remove.
  • Watchlist tab with snapshot summary rows.
  • localStorage save and restore for watched snapshots.

Excluded

  • No backend account storage.
  • No live refetch in watchlist rows.
  • No expanded watchlist cards.
  • No remove control inside the watchlist view; Story 2.3 owns that.
  • No charts, search history, or multi-list organization.

Data model

Type Fields Rule
WatchlistItem symbol, companyName, currentPrice, dollarChange, percentChange Snapshot copied from selected StockData.
WatchlistState items: WatchlistItem[] Stored as JSON under versioned key diamante.watchlist.v1.
Identity symbol Case-normalized unique key. Latest add updates existing snapshot.

Architecture

  1. Add src/types/watchlist.ts for the snapshot value object.
  2. Add pure helpers for toWatchlistItem(stock), upsertWatchlistItem(items, item), and removeWatchlistSymbol(items, symbol).
  3. Add useWatchlist() to load, validate, update, and persist localStorage-backed items.
  4. Validate parsed storage shape, not just JSON syntax; ignore rows missing required typed fields.
  5. Lift app-level tab and watchlist state into StockSearch unless the file would exceed limits; extract StockDiscoveryShell if needed.
  6. Pass isWatched and onToggleWatchlist into StockResult.
  7. Add WatchlistView and WatchlistItemCard for snapshot rows.
  8. Keep all formatting helpers small and shared only when a second component needs the same behavior.

Execution sequence

RED Acceptance first

  1. Add Story 1.3 acceptance tests for add, active state on re-search, remove toggle, and watchlist tab display.
  2. Use visible test data per docs/rules/TESTING-RULES.md; keep helpers local and DAMP.
  3. Reset localStorage before each watchlist test so persistence assertions do not leak between cases.
  4. Predict compile failure before adding new hook/component contracts.

GREEN Minimal implementation

  1. Create watchlist types and pure list operations.
  2. Implement localStorage hook with corrupt-storage fallback to empty list.
  3. Add tabs and watchlist view using snapshot rows.
  4. Add stock-card watch chip and status message.
  5. Wire selected stock toggle to latest-snapshot upsert and remove.

TIDY Refactor after green

  1. Check file length, function length, SRP, and duplicated formatting logic.
  2. Extract shell, tabs, or formatter modules only if limits or clarity demand it.
  3. Keep Story 2 behavior out of the refactor.

Testing plan

  1. Acceptance: add AAPL from card, see ★ Watching, see Added AAPL to watchlist.
  2. Acceptance: return to Search, load AAPL again, control starts active, click removes, see removal status.
  3. Acceptance: navigate to Watchlist tab and verify snapshot row includes symbol, company, price, dollar change, percent change.
  4. Acceptance: after removing AAPL, navigate to Watchlist tab and verify AAPL no longer appears.
  5. Unit: pure upsert updates existing symbol instead of duplicating.
  6. Unit: localStorage hook restores valid items, ignores invalid JSON safely, and filters malformed item rows.

Quality gates: npm test · npm run type-check · npm run lint · npm run prettier:check · npm run build

Manual verification

  1. Search AAPL, select preview, click Watch.
  2. Confirm chip changes to ★ Watching and status says Added AAPL to watchlist.
  3. Open Watchlist tab and confirm AAPL snapshot row appears.
  4. Search AAPL again and confirm active state is restored.
  5. Click Watching and confirm removal status plus empty watchlist state.
  6. Refresh browser and confirm localStorage-backed state survives.

Risks and mitigations

Risk Mitigation
Story 1.3 overlaps Epic 2 watchlist management. Build only minimal snapshot display and no watchlist-row remove control.
Stale watchlist prices look live. Plan labels them as snapshots and avoids refetch until Epic 2/3 decides live behavior.
localStorage parse errors break app load. Hook catches invalid storage, filters malformed rows, and returns empty list when no valid items remain.
Persistent watchlist tests leak state across cases. Clear localStorage in test setup and seed only the data each case needs.
Stock card file crosses project limits. Extract watch chip and status pieces before adding more branches.
